# Baseline measurement — v0.3.0-rc1

**Date**: 2026-05-20
**Spec**: `docs/PRINCIPLES.md` §3 (Baseline anchor 영구화)
**Tool**: `python -m ipe.baseline batch`

---

## 1. 목적

IPE multi-mechanism (M1 Sub-agent + M2 Hook + M3 Multi-model consensus + M4
Adversarial review) 의 quality 가치를 정량 검증.

비교 대상: **단일 LLM call** (1 Opus call이 problem + sample + solution 통합
생성). IPE pipeline (architect → designer → coder → reviewer → executor, 12-24
LLM calls per run) 와 같은 5 algorithm anchor 측정.

---

## 2. 측정 조건

- **Model**: `claude-opus-4-7` (IPE 의 architect 와 동일)
- **Algorithms**: Two Sum / BFS / Dijkstra / LIS / Segment Tree
- **N per algorithm**: 1 (잠정 — N=3 추가 측정은 후속 PR)
- **Verification**: sample 만 (Phase B/C generator/adversarial 은 IPE 만의 검증
  layer, baseline scope 밖)
- **Sandbox**: `RlimitRunner` (단순 sample run, 5초 cap)

## 5. 해석

### 5.1 단일 LLM 의 약점

LLM 이 자기 솔루션 mental simulation 결과를 "expected" 로 적는 과정에서 오류
발생. 특히:
- **BFS**: 3D state (격자 + 텔레포트) — head 시뮬레이션 어려움
- **LIS**: DP 추적 실수
- **Segment Tree**: range query 한 케이스에서 산술 오류

이건 IPE 가 fix 하려고 한 정확한 문제 — sample 실행으로 wrong 발견 가능.

### 5.2 IPE 의 약점

IPE 가 wrong sample 을 발견은 함 (Phase A mismatch). 그러나 **budget 안에 회복
못 함** — coder retry 4번에도 같은 알고리즘 misunderstanding 반복. 또 LIS 에서
M3 dual-call architect 가 둘 다 invalid → 0 iter 소진 (M3 도입의 부작용).

### 5.3 결론

**multi-mechanism 의 가치 명확히 입증 안 됨**:
- run-level: baseline win (2:1)
- sample-level: IPE win (90% vs 79%) but 운영적 의미 약함

가능한 해석 3가지:
1. **N=1 noise** — 더 측정 필요 (룰 1 적용)
2. **IPE의 information bottleneck**이 sample 검증 가치보다 큰 cost
3. **budget tuning** 만 하면 IPE win 가능 (coder=4 → 6, max-iter=12)

## 6. 다음 단계

### 6.1 N=3 추가 측정 (룰 1 적용)

baseline N=3 (15 runs) + IPE N=3 추가 (10 runs) → 25 runs 측정. 비용 측면:
- baseline 15 runs × ~$0.05 each = ~$0.75
- IPE Phase 2 10 runs × ~$0.5 each = ~$5
- 총 ~$6 (수용 가능)

### 6.2 v0.3.0 release 판정

baseline N=3 vs IPE N=3 비교:
- **baseline ≫ IPE**: M1/M3/M4 일부 rollback (PRINCIPLES.md §3 적용)
- **baseline ≈ IPE**: IPE 는 검증 layer (Catalog / Replay / Phase B/C) 만 정당화
- **baseline ≪ IPE**: 현 방향 유효, budget tuning 후 tag

### 6.3 알려진 한계 (이 측정의)

- **N=1**: variance 크다. 같은 algorithm 다시 돌리면 결과 다를 가능성 ↑.
- **Parser fix 직후 측정**: baseline runner v1 에서 unparseable 3개 발생 후 v2
  (brace-balanced) 로 fix 한 직후 측정. parser 의 quality 가 baseline 의 quality
  를 가린 가능성. v2 측정만 이 보고서에 사용.
- **Sample 만 검증**: Phase B/C 는 IPE 만의 가치 layer. baseline 비교는 sample
  까지만. full pipeline 비교는 별도 측정 필요.
