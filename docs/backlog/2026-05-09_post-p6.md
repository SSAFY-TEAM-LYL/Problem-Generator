## A2 — `tests/test_generator.py` 미작성

### 위치
`tests/test_generator.py` — 미작성. `ipe/nodes/generator.py` coverage 0%.

### 문제
`tests/integration/test_phase_c.py`는 결정론적 검증을 위해 generators 딕셔너리를
직접 `state["generators"]`에 inject한다 (LLM mock 없이). 따라서 `generator.run`,
`_parse` (`_BLOCK_RE` regex), `_route_back` (self-loop), `<MIN_GENERATORS` 분기,
feedback suffix 처리 등은 직접 검증되지 않는다.

### 미뤄둔 이유
- `_parse` regex는 P6.1 commit (724818e)에서 manual smoke test로 검증되었고,
  named capture group 사용으로 안정성이 높다.
- self-loop 패턴은 `architect.py` / `auditor.py`와 동일 패턴이며, 동일 패턴의
  단위 테스트는 `test_architect.py` / `test_auditor.py`에 이미 존재 — generator도
  같은 형태로 추가하면 됨 (보일러플레이트).
- P3 audit C2와 동일한 trade-off: 통합 테스트가 핵심 분기를 커버하나
  edge cases는 미검증.

### Resolution 방향
P9 (cost/history guard) 진입 시점 또는 P12 polish 라운드에서 `tests/test_generator.py`
추가. 패턴은 `tests/test_auditor.py`를 그대로 따른다:
- `_parse_*` 단위 테스트 (multiple blocks, malformed fence, partial match)
- self-loop on `<MIN_GENERATORS`
- feedback suffix on `state["feedback_message"]` 존재 시
- `MAX_GENERATORS` cap (6개 입력 → 5개로 잘림)

## A7 — full pipeline `graph.invoke` 통합 테스트 부재

### 위치
`tests/integration/` 안에 있는 모든 테스트는 노드를 직접 호출 (`auditor.run`,
`executor.run` 등). LangGraph `build_graph().invoke()`를 호출하는 e2e 사이클은 없음.

### 미뤄둔 이유
P3~P6는 직선 경로 (architect → coder → auditor → executor) — graph가 정의되어
있어도 conditional routing이 없어 의미 있는 사이클 테스트가 어렵다.

### Resolution 시점
**P7 (graph wiring + conditional routing) 진입 직후 즉시 처리.**
- happy path: 직선 통과
- coder failure → coder 재호출 (max iter 도달 후 abort)
- auditor failure → auditor self-loop
- generator failure → generator self-loop
