## B3 — Coder의 `tracker=None` 분기

### 위치
`ipe/nodes/coder.py` `run()` 함수 — `tracker is None`인 경우 `chat.invoke` 직접 호출.

### 문제
production path (`tracker` 주입)와 test path (`tracker=None`) 가 분기되어 있어,
test에서는 통과하지만 production에서만 발생하는 버그가 잠복할 수 있다.
LLMCallTracker가 `state["llm_calls"]`에 기록을 누적하므로 두 경로의 state 결과도 다르다.

### 미뤄둔 이유
P3은 "최소 회로" 단계이고, P4부터 Architect/Auditor/Generator/Evaluator가
모두 같은 패턴을 사용한다. **모든 노드가 등장한 후 한 번에 통일**하는 것이
일관적이고, 그 시점에 `make_*_node(tracker)` 팩토리 패턴 또는
`graph.py`의 `partial` 주입으로 정리 가능.

### Resolution 방향
- Option A: `tracker`를 required (no default) — 모든 노드가 강제. test는 mock tracker 주입.
- Option B: `make_coder_node(tracker)` 팩토리 도입. graph.py가 host. closure로 주입.
- 둘 중 P4 시작 시점 결정.

## B5 — sandbox-exec의 `(allow process-exec)`

### 위치
`ipe/sandbox/sandboxexec_runner.py` `_build_profile()` 의 sandbox profile.

### 문제
T2.5 macOS 프로파일은 `(allow default)` + targeted deny 전략이라
`process-exec`가 암묵적으로 허용된다. 이로 인해 sandbox 안에서
`/usr/bin/curl` 같은 binary를 직접 호출 가능 (network deny로 일부 완화되지만,
local socket 또는 파일 IPC를 거치는 attack vector가 남음).

### 미뤄둔 이유
- macOS sandbox-exec는 Apple deprecated. 정책 강화의 한계가 있음.
- 운영 환경은 T1 (Docker)을 권장하며, 그 경우 `--read-only` + `--network=none`로 binary 호출이 무력화됨.
- T2.5는 dev 환경 fallback 위치이므로 보안 약화는 알려진 trade-off.

### Resolution 방향
- 옵션 1: profile에 `(deny process-exec*)` 추가 후 필요 binary만 named allow — 호환성 깨질 위험 큼.
- 옵션 2: T2.5를 deprecated로 표시하고 macOS도 Docker 강제 권장 (CHANGES + README 업데이트).
- P12 (보안 리뷰)에서 결정.

## C2 — `tests/test_coder.py` 단위 테스트

### 위치
`tests/test_coder.py` — 미작성.

### 문제
`_parse_response()`의 fence 우선순위 / IMPOSSIBLE 검출 로직이 통합 테스트
(`test_minimal_circuit.py`)에서 일부 커버되지만, edge cases (다국어 펜스 라벨,
escape된 백틱, 빈 펜스 등)는 검증되지 않음.

### 미뤄둔 이유
- 통합 테스트의 4 시나리오에서 happy path / IMPOSSIBLE / wrong-output / syntax-error가
  각각 `_parse_response`의 핵심 분기를 1번씩 통과한다.
- 단위 테스트의 가치는 P4에서 architect.py가 추가될 때 동일 패턴
  (`tests/test_architect.py` + `tests/test_coder.py`) 으로 일관 도입하는 것이 효율적.

### Resolution 방향
- P4 architect 도입 시 `test_architect.py` + `test_coder.py` 동시 작성.
- 또는 Phase B/C 추가 시 (P5/P6)에 노드별 단위 테스트 일괄 정비.

## D2 — Phase A만으로 `success` (의도된 한계)

### 위치
`ipe/nodes/executor.py` `run()` 끝 부분 — Phase A 통과 시 즉시
`final_status="success"`를 set.

### 영향
P5 (Phase B 추가) / P6 (Phase C 추가) 전까지 end-to-end 사이클이
"Phase A만 통과한 partial 검증"으로 success 표시. **false positive 주의**.

### Resolution 시점
- P5: Phase B 추가 시 success 조건에 Phase B all-pass 추가
- P6: Phase C 추가 시 success 조건에 Phase C all-pass + 정해 성능 게이트 추가
- 최종 `final_status="success"`는 3-Phase 모두 통과 + 성능 게이트 통과 시에만 set

### 임시 안전망
README.md / CHANGES.md에 "P5/P6 완료 전까지 success는 부분 검증" 명시 (P3 완료 시점에 이미 README 수정됨).
