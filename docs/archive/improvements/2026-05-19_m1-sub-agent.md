## 1. 동기

### 1.1 Coder 단일 노드의 책임 과부하

기존 Coder가 1 LLM call에 처리하는 책임:
1. **Algorithm 선택**: 문제 보고 BFS/DP/segment tree 등 결정
2. **Implementation**: Python/Java 코드 작성 (I/O 최적화 + edge case)
3. **Brute solution**: R15 cross-check용 naive 구현
4. **LESSON**: R13 1-line reflexion (이전 실패 학습)

LLM이 모든 책임을 동시에 잘하기 어려움 — 어느 하나가 약하면 quality ↓.
특히 Round 17~19 측정에서 "oracle slow" / "Phase A mismatch" 패턴이 자주 발견 →
algorithm 선택 부정확이 근본 원인 가능성.

### 1.2 ECC subagent 패턴

ECC는 책임 좁은 specialized agent를 chain — `code-explorer` (research) +
`coder` (implementation) + `code-reviewer` (verify) 등 분리. 각 agent가 좁은
도메인에 집중하여 quality ↑.

IPE에 적용: **Coder를 AlgorithmDesigner + Coder로 분해**.
- AlgorithmDesigner: problem → algorithm 결정 + pseudocode
- Coder: pseudocode → 구현 (algorithm 선택 부담 X)

## 2. 설계

### 2.1 그래프 topology 변화

```
Before:
  START → architect → coder → executor → decision → ...

After:
  START → architect → algorithm_designer → coder → executor → decision → ...
```

`decision` 노드의 conditional_edges에 `algorithm_designer` 추가 — designer가
fail하면 자체 self-loop, budget 소진 시 `budget_exhausted` halt.

### 2.2 AlgorithmDesigner 노드

`ipe/nodes/algorithm_designer.py`:

```python
SYSTEM_PROMPT = """You are The Algorithm Designer — a master algorithm strategist.

Given a problem statement, constraints, and sample testcases, decide:
1. Algorithm name (BFS, DP, segment tree, etc.)
2. Pseudocode (5-15 lines, language-agnostic)
3. Complexity target (Big-O time + space)
4. Edge cases (3-7 specific cases)

Output JSON wrapped in fence.

DO NOT write actual code — that's the Implementer's job.
Choose algorithm that fits time/memory constraints. If N=200000 and time=2s,
O(N^2) is wrong; pick O(N log N) or O(N).
"""
```

**Model**: Sonnet (`DESIGNER_MODEL = "claude-sonnet-4-6"`) — algorithm 선택 +
pseudocode만이라 Opus 수준 reasoning 불필요 + cost ↓ + latency ↓.

**temperature**: 0.3 — 결정적 design 우선 (변동성보다 정확성).

## 3. ECC mapping

| ECC primitive | IPE M1 구현 |
|---|---|
| Subagent (책임 분리) | architect (problem) + algorithm_designer (algorithm) + coder (implementation) |
| `code-explorer` agent | AlgorithmDesigner — 분석 + 설계 |
| `coder` agent | Coder — 구현에 집중 |
| Multi-step planning | architect → designer → coder 순차 책임 |

## 5. 한계 + 후속

- **designer 실패 routing 정책**: 현재 designer가 fail하면 self-loop. 만약 designer
  oscillation 발생하면 budget 소진 → halt. R-osc-break의 swap 매핑에 designer 추가
  검토 가능 (designer 실패 → architect로 escalate — problem이 unclear한 가능성).
- **algorithm_design quality 측정**: e2e run에서 designer가 정확한 algorithm 선택하는지
  metric 필요 (현재는 수동 trace 분석).
- **prompt 비용 증가**: coder prompt에 algorithm_design block 추가 → input tokens
  20-30% ↑ 예상. cost vs quality tradeoff 측정 필요.
- **e2e 측정 필요**: M1 적용 후 BFS/SegTree e2e — designer가 algorithm 명시한 후
  Coder의 implementation 품질 변화 확인.
