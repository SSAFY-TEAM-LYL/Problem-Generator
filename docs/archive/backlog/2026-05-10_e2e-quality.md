### 🟡 W4 oscillation 경고가 LLM에 무력함

`build_history_section`이 동일 (node, error_signature) 2회 시 "DIFFERENT STRATEGY REQUIRED — Do NOT repeat the same approach; switch to a fundamentally different strategy" 경고를 user prompt에 자동 삽입하지만, **Coder가 똑같은 fix 반복**:
- Two Sum `f16cb8e1`: iter 3, 5, 6 (3회)
- Dijkstra `178582d4`: iter 4, 5, 7 (3회)

LLM이 "다른 전략" 지시를 prompt에서 받았음에도 같은 RTE/TLE 패턴 재현 → **prompt-only 강제는 효과 제한적**.

LIS만 Coder까지 도달도 못 함. iter 3에서 `phase C: 5 generator scripts failed (generator must regenerate)` → generator self-loop → budget=2 소진. **Generator LLM이 LIS 입력 형식 (DP 시퀀스) 생성 script를 못 작성** — 별개 문제.

### 인사이트 (Run 2 추가)

**1. Budget 상향만으로는 0/5 → 0/5** — 본질적 해결 X

- BFS / Segment Tree / LIS는 `iter=9`까지 도달 (이전 7) — Coder가 budget 6번 다 쓰고도 stress case 통과 못 함
- **Coder의 fix attempt가 누적 학습 안 됨** — W4 경고에도 동일 패턴

**2. LIS는 `generator_budget 2→3` 효과 입증**
- Run 1: Generator 5/5 script fail → 즉시 종료
- Run 2: Generator 통과 → Coder까지 도달 → iter 3-8 모두 coder
- generator는 budget 상향이 효과적, **coder는 효과 없음**

**3. Two Sum 악화 — Auditor 비결정성 부각**
- Run 1: auditor 1번에 valid → coder까지 도달
- Run 2: auditor가 `violate constraints` 입력 생성 → 2회 self-loop → halt
- **`auditor_budget=2`가 LLM 분산을 못 흡수**

### 권장 다음 단계

**구조적 개선 (v0.2.0 후보)**:

1. **Coder 단일-shot 한계 보완**:
   - `--coder-fanout N`: 한 cycle에 N개 다른 implementation 동시 생성, executor가 best 선택
   - 또는 Coder 안에서 self-critique pass (작성 → 자체 review → 수정)
2. **W4 oscillation 강화**:
   - 동일 시그니처 3회+ → **architect로 라우팅** (문제 재설계)
   - 또는 `temperature` 증가 (현재 default 0.7)
3. **Auditor budget 자동 상향** — `auditor_budget`을 default 4 (LLM 분산 흡수)
4. **Brute-force cross-check** (ROADMAP §7):
   - Coder가 golden + brute 둘 다 작성
   - Executor가 동일 stress 입력에서 두 출력 비교 → fail 시 어느 쪽이 틀렸는지 추적
5. **Cost 측정 검증** — Anthropic console과 일치 여부 정확히 측정 (cache_creation/read 토큰 처리)
