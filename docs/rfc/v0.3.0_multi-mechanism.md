**Status**: Draft

## 0. 한 줄 요약

ECC-style 다층 메커니즘 (Sub-agent + Hook + Multi-model + Adversarial-review)을
single-LLM-call 파이프라인 위에 layer로 쌓아 stochastic e2e success를
결정론적 5/5에 가깝게 끌어올린다. 한 PR에 한 메커니즘씩 incremental,
매 PR 후 즉시 e2e 측정.

---

## 1. 동기

### 1.1 Round 11~19 측정으로 드러난 본질

| Round | Fix | 작동 | e2e 효과 |
|---|---|---|---|
| 11~18 (9 fix) | oscillation breakers + infra + retry | unit/integration 모두 ✓ | SegTree 1회 success (variance), BFS 여전히 fail |
| **19 (R5)** | brute oracle Phase A | ✓ | **BFS Docker success** (1회) — sample-wrong 결정적 차단 |

R5로 BFS success 첫 도달. 그러나:
- **Single run success ≠ 결정적**. LLM 응답 variance가 본질적 — 단일 LLM call 위에 fix가 쌓여도 분포만 좁아질 뿐 5/5 보장 못 함.
- 매 Round 측정에서 새 fail mode 발견 (sample-wrong → oscillation → oracle slow → fenced parse → ...). 하나 잡으면 다른 게 노출되는 long tail.

### 1.2 ECC mapping — 왜 multi-mechanism인가

ECC (Everything Claude Code)는 단일 Claude LLM call의 stochasticity를 보완하기 위해 **여러 작은 메커니즘이 협력**:

| ECC primitive | 의미 | IPE 대응 (현재) | IPE 갭 |
|---|---|---|---|
| Rules | 절대 원칙 (coding-style, testing, ...) | 노드별 SYSTEM_PROMPT | ✓ 있음 |
| Skills | 도메인별 specialized procedure | (없음) | **갭** — algorithm 카테고리별 reference 없음 |
| Subagents | 책임 좁은 specialized agent | 6 노드 (architect/coder/...) | **갭** — Coder 한 노드가 알고리즘 설계 + 구현 다 함 |
| Hooks | pre/post tool-use mandatory check | (없음) | **갭** — Phase 진입 전 정적 분석 없음 |
| MCP servers | 외부 capability | RlimitRunner/DockerRunner + brute oracle (R5) | ✓ 부분 |
| Multi-model | 모델 다양화 | (없음, 단일 모델 per 노드) | **갭** — Architect/Coder 단일 모델 |

**ECC의 핵심**: 한 메커니즘이 missed → 다른 메커니즘이 catch. **IPE는 단일 LLM call의 single point of failure** — fix가 쌓여도 LLM 본질적 variance 영향.

### 1.3 R5가 가르쳐준 패턴

R5는 brute solution을 oracle로 사용 — **단일 LLM call 결과를 "다른 결정적 메커니즘"으로 cross-check**하는 첫 사례. 효과 직접 확인.

이 패턴을 일반화: **모든 핵심 phase에 cross-validation 메커니즘 도입**.

## 2. 4종 메커니즘 design

### M1 — Sub-agent 분해 (Coder → AlgorithmDesigner + Implementer)

**문제**: Coder 한 노드가 (a) 알고리즘 선택 + (b) Python 구현 + (c) brute solution + (d) LESSON 생성을 한 LLM call에 처리. 책임 분산 → quality 낮음.

**해법**:
```
[Coder 분해]
AlgorithmDesigner: problem + constraints → algorithm name + pseudocode + complexity analysis
   ↓ (algorithm.json: {name, pseudocode, complexity_target, edge_cases})
Implementer: algorithm.json + problem → Python solution + brute solution
   ↓ (solution_code + brute_solution_code)
```

| 측면 | 변화 |
|---|---|
| state schema | `+algorithm_design: dict` (designer 출력) |
| 그래프 노드 | `architect → algorithm_designer → implementer → executor` |
| 단위 PR scope | 1 PR (Coder 분해 + 신규 designer 노드 + 라우팅) |
| 회귀 risk | medium — graph topology 변경 |
| 측정 | BFS/SegTree e2e × 2회 |

**ECC 직접 mapping**: `code-explorer` (Designer) + `coder` agents 분리 패턴.

### M2 — Hook-driven pre-verification (Phase 진입 전 정적 분석)

**문제**: Phase A/B/C가 LLM 응답을 일단 실행하고 fail 시 routing. 명백한 invalid 케이스 (input format mismatch, complexity가 N에서 무조건 TLE 등)도 실행 비용 지불.

**해법**: Phase 진입 전 정적 checker가 mandatory verify.

```python
@hook("pre_phase_b")
def check_complexity_feasibility(state):
    """time_limit_ms와 N_max로 허용 가능한 complexity 추정.
    O(N^2) for N=200000 → 즉시 reject."""
    ...

@hook("pre_executor")
def check_solution_imports(state):
    """solution_code에 표준 라이브러리 외 import 있으면 reject."""
    ...
```

| 측면 | 변화 |
|---|---|
| state schema | (없음, hook은 state read-only) |
| 그래프 노드 | (없음, hook은 노드 진입 wrapper) |
| 단위 PR scope | small — hook 인프라 + 2-3 hook 함수 |
| 회귀 risk | low — additive |
| 측정 | hook fire rate measurement |

**ECC 직접 mapping**: `PreToolUse` hook 패턴.

### M3 — Multi-model consensus (Architect Opus + Sonnet voting)

**문제**: Architect 출력 (problem + sample expected)이 single Opus call. 모델 hallucination 가능 (BFS sample expected 오류 사례).

**해법**:
```
Architect (Opus + Sonnet 병렬 호출)
   ↓
Voting:
  - Opus problem == Sonnet problem (structural diff) → consensus → 진행
  - Diff 있음 → consensus 실패 → 재시도 (Sonnet 비중 ↑) 또는 architect retry
```

| 측면 | 변화 |
|---|---|
| state schema | `+architect_candidates: list[dict]`, `+consensus_status: str` |
| 그래프 노드 | architect 노드가 2 LLM call 병렬 |
| 단위 PR scope | medium — voting logic + 비용 측정 |
| 회귀 risk | medium — cost 2×, latency 거의 동일 (병렬) |
| 측정 | consensus_rate measurement |

**ECC 직접 mapping**: `santa-loop` adversarial pattern의 generator 측.

### M4 — Adversarial dual-review (Solution → Reviewer gate)

**문제**: Coder가 만든 solution을 Executor가 sample 실행만으로 검증. complexity / edge case / 코드 quality는 검증 안 됨.

**해법**:
```
Implementer → solution_code
   ↓
Reviewer (별도 LLM call): "이 solution이 problem을 정확히 풀고 edge case를 cover하는가?"
   - approval → Executor 진입
   - rejection → reasoning을 Implementer feedback에 주입 → 재시도
```

| 측면 | 변화 |
|---|---|
| state schema | `+review_status: str`, `+review_reasoning: str` |
| 그래프 노드 | `implementer → reviewer → executor` |
| 단위 PR scope | medium |
| 회귀 risk | medium — 1 cycle = 2 LLM call (Implementer + Reviewer) |
| 측정 | reviewer rejection rate, false-positive rate |

**ECC 직접 mapping**: `santa-loop`의 reviewer 측 또는 `code-reviewer` agent.

## 3. 의존성 + sequencing

```
       (현재 main: R5 머지 + v0.2.1 release)
                    │
                    ▼
   [M2 Hook-driven pre-verification]  ← 1순위
       (가장 작음, additive, 다른 메커니즘 기반 인프라)
                    │
                    ▼
      [M1 Sub-agent 분해 (Coder)]  ← 2순위
       (Designer 분리는 M3/M4의 dependency)
                    │
       ┌────────────┴────────────┐
       ▼                         ▼
[M3 Multi-model consensus]  [M4 Adversarial review]
       (Architect 측)           (Implementer 측)
                    │
                    ▼
              [v0.3.0 release]
                e2e DoD 측정
```

**근거**:
- M2가 가장 작음 + 다른 메커니즘과 독립적 → 안전한 첫 PR
- M1 (Designer 분리)이 M3/M4의 input — algorithm_design field를 Reviewer가 cross-check
- M3와 M4는 병렬 가능 (서로 독립적). M3는 Architect, M4는 Implementer 측

**순차 (1 PR씩)**:
1. M2 Hook infra + 2-3 hook
2. M1 Sub-agent (Coder 분해)
3. M3 Multi-model consensus (Architect voting)
4. M4 Adversarial review (Reviewer gate)
5. v0.3.0 release

각 PR 머지 후 즉시 BFS + SegTree e2e 1회씩 측정. Round 11~19 패턴 그대로.

## 4. 측정 계획

### 4.1 매 PR 후 measurement

| 단계 | 측정 | 목적 |
|---|---|---|
| PR 머지 | BFS Docker × 1 + SegTree Docker × 1 | smoke + 회귀 0 |
| 4 PR 모두 머지 | BFS × 3 + SegTree × 3 (consecutive) | variance + DoD |

### 4.2 단위 metric

- **e2e success rate**: 5/5 cases × 3 runs = 15 attempts 중 success%
- **iteration_count 평균**: 효율 측정 (낮을수록 좋음)
- **cost USD per success**: $ per success run
- **hook fire rate** (M2): hook이 catch한 케이스 / 전체
- **consensus rate** (M3): Opus-Sonnet 일치 / 전체 architect call
- **reviewer rejection rate** (M4): Reviewer reject / 전체 implementer call

### 4.3 회귀 detection

각 PR 후:
- 전체 pytest 회귀 0
- 이전 measurement metric에서 ±20% 이상 변동 시 RCA

## 5. v0.3.0 Release DoD

| Criterion | Threshold |
|---|---|
| e2e success rate | **>=80%** (5 cases × 3 runs = 15 중 12+ success) |
| iteration_count 평균 | <=8 (v0.2.0 baseline ~7-12) |
| 회귀 | 0 |
| ruff / mypy --strict | 0 / 0 |
| 신규 tests | +50 minimum |
| RCAs | 4 (메커니즘별 1개) |
| Cost per run | <=$3 (M3 voting으로 cost 2× 우려 — 측정 후 결정) |

**Release 보류 trigger**:
- e2e success rate < 60% → 메커니즘 design 재검토
- 회귀 > 0 → 즉시 fix
- Cost > $5 per run → M3 model 조정 (Sonnet 비중 ↑)
