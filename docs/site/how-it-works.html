실제 코드 포함 개발자도 읽을 만한 깊이

IPE는 어떻게
문제를 만들고 검증할까?

비유로 큰 그림을 잡고, 실제 소스 코드로 "정말 그렇게 동작하는지" 확인합니다.
핵심 한 문장 — "AI가 만든 답이 맞는지, 코드가 수학으로 다시 계산해 검산한다."

한 줄로 말하면

IPE는 알고리즘 코딩 문제를 찍어내는 공장입니다. 그냥 찍기만 하면 불량품(정답이 틀린 문제)이 섞이겠죠. 그래서 공장 끝에 "불량 검사기" 를 달았는데, 이 검사기가 사람·AI의 감(感)이 아니라 알고리즘의 수학 정의로 답을 직접 다시 계산해서 통과시킵니다. 아래에서 그 검사기의 실제 코드를 보여드릴게요.

1. 왜 이게 어려운 일일까?

"AI한테 코딩 문제 하나 만들어 줘" 라고 하면 그럴듯한 문제가 나옵니다. 문제는 그 문제의 "정답"이 진짜 정답인지 아무도 모른다는 거예요.

예: AI가 "도시 5개가 도로로 연결돼 있을 때 A→B 최단 경로 길이는?" 문제를 만들고 정답을 "17" 이라 적었습니다. 이 17이 맞을까요? AI는 글은 잘 쓰지만 계산은 틀리기도 합니다. 정답이 틀린 시험 문제를 응시자에게 내면 채점이 통째로 어긋납니다.

🍪 비유

쿠키 공장이 모양 예쁜 쿠키를 잘 굽는다고 안에 든 게 다 안전한 건 아니죠. "먹어도 되는지" 는 따로 검사해야 합니다. AI가 쓴 문제도 똑같아요 — 지문(모양)이 좋아도 정답(속)이 맞는지는 별도로 검사해야 합니다.

2. 핵심 아이디어 — "AI에게 다시 묻지 않고, 계산기로 검산한다"

여기가 IPE의 심장입니다. 정답이 맞는지 확인할 때 다시 AI에게 물어보지 않습니다. 대신 그 알고리즘의 수학 정의를 코드로 짜 두고(이걸 검증기 verifier 라고 불러요), 그 코드가 답을 처음부터 다시 계산합니다.

"최단 경로 17" 문제를 봅시다. 최단 경로 검증기에는 4가지 수학 법칙(invariant)이 박혀 있어요. 코드 주석에 그대로 적혀 있습니다:

ipe/v1/verifiers/dijkstra.py
# Dijkstra symbolic verifier — 4 invariants 결정론적 검증
#
# 1. non_negative_distance : 거리는 음수일 수 없다
# 2. source_zero           : 출발=도착이면 거리는 0
# 3. reachability_consistent: 도달 불가인데 거리값이 있으면 모순
# 4. shortest_distance_optimal: 독립 알고리즘(Bellman-Ford)으로
#    다시 계산한 답과 일치해야 한다  ← 진짜 검산

네 번째가 핵심입니다. 문제를 만든 AI(이건 Dijkstra로 풀었다고 칩시다)와 완전히 다른 알고리즘인 Bellman-Ford로 답을 독립적으로 다시 계산합니다. 두 답이 다르면? AI가 틀린 거예요.

ipe/v1/verifiers/dijkstra.py — 검증기가 직접 답을 다시 계산
def _shortest_distance_bellman_ford(graph):
    # 출발점 거리는 0, 나머지는 무한대로 시작
    dist = [inf] * graph.V
    dist[graph.s] = 0.0
    for _ in range(graph.V - 1):
        for u, v, w in graph.edges:
            # 더 짧은 길을 찾으면 거리 갱신
            if dist[u] != inf and dist[u] + w < dist[v]:
                dist[v] = dist[u] + w
    return int(dist[graph.t])   # s→t 최단 거리
ipe/v1/verifiers/dijkstra.py — AI의 답(actual) vs 검산값(golden) 대조
# invariant 4: 우리가 다시 계산한 답과 비교
golden = _shortest_distance_bellman_ford(graph)
if actual != golden:
    return InvariantViolation(
        invariant_kind="shortest_distance_optimal",
        description=f"sample {i}: actual={actual} != "
                    f"Bellman-Ford golden={golden}",
        ...
    )

→ 만약 AI가 17, 검증기가 15를 계산했다면 actual=17 != golden=15 로 즉시 불합격 처리됩니다. 사람 판단도, AI 감도 개입하지 않아요.

💡 왜 이게 강력한가

검증기는 문제의 지문(글)을 읽지 않습니다. 위 코드처럼 오직 그래프 데이터와 알고리즘 수학 정의에서 답을 끌어냅니다. 그래서 문제를 만든 AI가 지문을 잘못 이해해 헛소리를 적어도, 검증기는 거기에 휘둘리지 않아요. "글을 오해할 위험"과 "검산"이 코드 레벨에서 분리돼 있다는 게 핵심입니다.

이런 검증기가 현재 19개 알고리즘(최단 경로·정렬·최소 신장 트리·배낭 등)에 준비돼 있고, 측정 결과 만든 문제의 약 91%가 검증을 통과합니다. 통과 못 하면 버립니다 — "맞다고 확인된 문제만 내보낸다" 가 원칙이에요.

3. 일꾼들은 "글"이 아니라 "타입이 정해진 서류"로 대화한다

문제 하나는 여러 AI 일꾼의 손을 거칩니다(다음 장에서 봅니다). 이때 일꾼들이 서로 자유로운 문장으로 소통하면, 받는 쪽이 해석을 또 틀립니다. 그래서 IPE는 일꾼 사이 통신을 형식이 고정된 서류(typed contract)로 강제합니다.

예를 들어 "검사 결과"는 자유 문장이 아니라, 실패 종류가 미리 정해진 목록(enum)으로만 표현됩니다:

ipe/v1/schema/verification_result.py
class FailureMode(StrEnum):
    NONE                = "none"                # 통과
    SAMPLE_MISMATCH     = "sample_mismatch"     # 예시 답이 안 맞음
    INVARIANT_VIOLATION = "invariant_violation" # 수학 법칙 위반
    BRUTE_DISAGREEMENT  = "brute_disagreement"  # 정해 vs 브루트 불일치
    SAMPLE_CRASH        = "sample_crash"        # 코드가 터짐
    ...

class StructuredFeedback(BaseModel):
    target_node:        TargetNode  # 누구에게 돌려보낼지 (enum)
    actionable_hint:    str         # 무엇을 고칠지
    blocking_signature: str         # 같은 실패 반복 감지용 지문
📋 비유

자유 메모("음… 뭔가 좀 이상한데요?") 대신, 체크박스가 정해진 검사 양식을 쓰는 거예요. "실패 사유: ☑ 예시 답 불일치 / 담당: ☑ 출제자 / 메모: 예시 3번 답 확인 요망." 받는 쪽이 해석할 여지가 없으니 다음 행동이 기계적으로 정해집니다. 이게 IPE가 거의 한 번에(평균 1.07회) 문제를 완성하는 비결이에요.

4. 문제 하나가 만들어지는 과정 (조립 라인)

문제는 컨베이어 벨트의 부품처럼 네 일꾼의 손을 거칩니다. 각자 한 가지만 잘해요.

1
출제자 (Architect)

"최단 경로" 키워드를 받아 이야기 있는 지문 + 예시 입력·출력을 씁니다. 결과물은 자유 글이 아니라 ProblemSpec 이라는 타입 고정 서류로 나와요.

↓
2
설계자 (Algorithm Designer)

이 문제를 어떤 방법으로 풀지 계획합니다 — 알고리즘 선택, 시간복잡도, 주의할 함정(엣지 케이스). 설계도를 그리는 단계.

↓
3
프로그래머 (Coder)

설계도대로 정답 코드 두 벌을 짭니다 — 빠른 "정해(golden)"와 느리지만 확실한 "브루트포스(brute)". 일부러 두 벌을 만들어 나중에 서로 답을 대조합니다.

↓
4
검사기 (Executor + Verifier) 핵심

2장의 검산이 여기서 돌아갑니다. AI가 만든 코드를 그냥 실행하면 위험하니 격리된 공간(sandbox)에서만 돌립니다. 그리고 수학 검증 + 정해↔브루트 대조로 합격/불합격을 판정합니다.

검사기가 똑똑한 진짜 이유는 "불합격"을 외치고 끝이 아니라, 누구 잘못인지 콕 집어 돌려보낸다는 점입니다. 실제 라우팅 코드를 보세요 — 우리가 측정으로 발견한 패턴이 그대로 코드에 박혀 있습니다:

ipe/v1/nodes/executor.py — 실패를 누구에게 돌려보낼지 결정
# 예시 답은 틀렸는데(SAMPLE_MISMATCH) 수학 invariant 는 전부 통과(violations 없음)?
# → 코드는 사실상 맞고, '출제자가 정답을 잘못 적은 것'이다.
# → 프로그래머가 아니라 출제자(ARCHITECT)에게 돌려보낸다.
if mode == FailureMode.SAMPLE_MISMATCH and not violations:
    target = TargetNode.ARCHITECT
    routed_hint = f"{hint} (verifier invariants all pass — "
                  f"likely architect expected_output error, regenerate spec)"
else:
    target = TargetNode.CODER   # 그 외엔 프로그래머가 코드 수정

→ "수학 검산은 다 맞는데 예시 답만 틀렸다"는 건 코드가 아니라 출제자가 예시 정답을 잘못 적었다는 강한 신호입니다. 그래서 출제자에게 되돌려 다시 쓰게 합니다. (이 한 줄짜리 규칙이 까다롭던 Two Sum 문제군을 회복시킨 실제 개선이에요.)

5. 한 발 더 — "검증기를 못 만드는 문제"는 어떻게 믿나

위 방식엔 한계가 있어요. 수학 검증기(Bellman-Ford 같은)는 "정석이 분명한" 19개 알고리즘에만 있습니다. 알고리즘을 숨기거나 두세 기법을 섞은 어려운 문제(= 진짜 기업 코테)는 정석 검증기가 안 맞아요. 그럼 그런 문제의 답은 어떻게 믿죠?

답은 정해(golden)와 브루트포스(brute)를 같은 입력에 돌려 답을 대조하는 겁니다 (ICPC·IOI 출제자들이 실제로 쓰는 방법). 두 코드가 같은 답을 내면 강한 신뢰 신호예요. 이게 지금 개발 중인 Phase 3 / v2의 검증 축(Tier B)입니다:

ipe/v1/verification/differential.py — 정해 vs 브루트 대조 (Phase 3)
# 같은 입력을 golden 과 brute 에 각각 실행
g = run_code(runner, golden_code, inp, ...)
b = run_code(runner, brute_code, inp, ...)

# 둘 다 정상 종료 + 출력이 같을 때만 '일치'로 인정
agreed = g.status == "OK" and b.status == "OK" and g_out == b_out
ipe/v1/verification/differential.py — "검사 안 했는데 통과"를 금지
@property
def all_agreed(self):
    # 입력이 0개면 False — 비교를 안 했으면 통과시키지 않는다.
    # (B2B 출하 정밀도: '조용한 통과'는 사고로 이어진다)
    return self.total > 0 and len(self.disagreements) == 0
⚠️ 진짜 위험은 "둘이 똑같이 틀리는 것"

정해·브루트가 지문을 똑같이 오해하면 둘 다 같은 오답을 내고 "일치"로 통과해 버립니다. 그래서 Phase 3는 두 코드를 서로 다른 출처에서 유도하고(정해=형식 명세, 브루트=서사 시나리오), 문제 종류별 불변관계(metamorphic) 검사를 더해 이 위험을 줄입니다. 그리고 문제마다 도달한 신뢰 등급(Tier A/B/C)을 매겨, 기업용은 Tier B 이상만 출하하고 미달은 폐기합니다.

6. "그래서 이 숫자들이 무슨 뜻이냐면"

사이트 곳곳의 숫자를 쉬운 말로 풀어 드릴게요.

91.2%
검증 통과율

만든 문제 100개 중 약 91개가 "정답이 맞다"고 검산을 통과. 처음(옛 버전)엔 27개뿐이었어요.

19
수학 검증기를 가진 알고리즘

최단 경로·정렬·배낭 등 19종은 코드가 답을 직접 검산할 수 있습니다(2장의 Dijkstra처럼).

99.1%
검사기가 실제로 일한 비율

"검사기가 있다"가 아니라 "거의 모든 예시에서 실제로 검산이 작동했다"는 뜻. 장식이 아니라는 증거.

1.07회
평균 시도 횟수

문제 하나를 완성하는 데 평균 1.07번. 3장의 typed 서류 + 4장의 정확한 라우팅 덕분에 거의 한 방에 끝납니다.

7. 지금은 어디까지 왔고, 어디로 가나

지금 버전(v1.0)은 위 과정을 안정적으로 해냅니다. 하지만 솔직히 말하면 아직 문제가 "쉬운 편"이에요. "이건 최단 경로 문제야"라고 알고리즘 이름을 대놓고 알려주고 시작하거든요. 실제 코테의 진짜 어려움은 "이 문제를 어떤 알고리즘으로 풀어야 하는지 스스로 알아내는 것"인데, 그게 빠져 있습니다.

🎯 다음 목표 (Phase 3 / v2)

다음 버전은 알고리즘 정체를 숨긴 현실 시나리오 문제를 만듭니다. "택배 기사가 가장 빨리 배송하려면…" 이야기 속에 최단 경로를 숨겨 두는 거죠. 대신 5장에서 본 검증 스택(differential + metamorphic + 신뢰 등급)으로 핵심 장치를 오히려 더 강화합니다. "믿을 수 있는 문제만 낸다"는 원칙은 절대 양보하지 않습니다.

모든 변경은 "91% 통과율을 떨어뜨리지 않는다" 는 측정 시험을 통과해야만 반영돼요 — 좋아지려다 나빠지면 즉시 되돌립니다.

📌 딱 세 줄 요약
  1. IPE는 코딩 시험 문제를 자동으로 만드는 기계다.
  2. 정답이 맞는지 AI에게 다시 묻지 않고, 알고리즘 수학 정의를 담은 코드로 직접 검산한다 (2장 Dijkstra 코드). 일꾼들은 타입 고정 서류로 통신하고(3장), 실패는 책임자에게 정확히 라우팅된다(4장).
  3. 검증기 없는 어려운 문제는 정해↔브루트 대조로 믿고(5장), 다음 버전은 "정체를 숨긴 진짜 시험 문제"로 간다. 검증 원칙은 끝까지 사수.
